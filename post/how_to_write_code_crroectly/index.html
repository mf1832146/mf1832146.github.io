如何生成可执行的代码？

2023-10-02 reads ， codegen

目前使用LM生成代码的方式是使用beam search或者sample解码。但是，这两种方式都无法保证生成代码的质量。并且，观察beam search生成的代码片段，发现其中出现大量无用重复的代码。由于分词的原因，代码生成属于长文本生成。beam在生成长文本时会出现大量重复(文本内和文本间)，以及无趣的情况。sample可以使得生成文本具有多样性，但是却也同时引入了随机性。那么该如何生成高可信，或者说可编译的代码是一个问题。

2.1 Planning with Large Language Models for Code Generation

在代码生成中融入强化学习的方法并不少见，比如CodeRL. 但这篇文章不同于之前的文章，将强化学习中的思想(decision making)融入到解码中，提出了一种基于蒙特卡罗树搜索(Monte Carlo tree search, MCTS)的解码方法。

  1. 选择 (Selection): 根据P_UCB算法选择最优节点进行扩展。P_UCB算法是结合探索(Exploit)和应用(Explore)的算法。

如何利用P_UCB选择一个最优节点？从root节点开始，依次选择子树,直到到达一个未被展开的节点为止。选择的策略倾向于选择1) 有较大Q(s,a)的节点 2) 模型认为较可能的动作a 3) s已经被探索过多次，但是动作a被选择的很少的节点. P_UCB公式如下.

P_UCB(s,a)=Q(s,a)+β(s)PTransformer(a∣s)log(s.visits)1+s′.visitsP\_UCB(s, a) = Q(s, a) + \beta(s) P_{Transformer}(a|s) \frac{\sqrt{log(s.visits)}}{1+s'.visits} P_UCB(s,a)=Q(s,a)+β(s)PTransformer​(a∣s)1+s′.visitslog(s.visits)​​

公式包含两部分。其中，sss表示当前已经生成的片段代码，aaa表示选择的下一个token. 左边的Q(s,a)Q(s,a)Q(s,a)即预期奖励，即未来能够通过编译或者测试用例的比率(文中称，使用该节点扩展的最优程序通过测试率最佳)。但是该奖励对于不完整代码都是0，且下一个词选择范围过多(等于词汇表大小)，因此, 增加了第二部分(探索)。首先规定，每一个状态sss下最多有kkk个动作，具体哪kkk个，由模型决定，即PTransformerP_{Transformer}PTransformer​. 其次，考虑到鼓励探索展开不充分的子树，因此，增加了log(s.visits)1+s′.visits\frac{\sqrt{log(s.visits)}}{1+s'.visits}1+s′.visitslog(s.visits)​​项. s.visits即s状态被选择的次数。s'是状态s通过a动作扩展后的新状态(将a拼接到s之后)。最后，是倾向于冒险探索，还是保守式地应用，由权重β(s)\beta(s)β(s)控制.

  1. 扩展 (Expansion): 通过模型选择最合适的kkk个动作aaa，将s′=(s,a)s' = (s, a)s′=(s,a)作为sss的子节点加入到树中.
  2. 评估 (Evaluation): 由于不完整的代码是不能去衡量它的价值(编译率，测试通过率), 因此评估时使用beam search直接解码出完整的代码，对这个节点的价值进行评估
  3. 更新 (Backpropagation): 对该节点及该节点的祖先节点价值进行更新，使用平均价值最为祖先节点的价值

思考:
虽然文章中提及了可以用缓存的方式避免重复计算。但是测试毕竟是一个比较耗时的方式，同时本文中的方法使用的存储资源应该不低。能否不通过测试的手段来获取到不完整代码的价值？可能的方案有A*算法. 通过思维链? 还是API使用范式？与代码的一致性关系，来计算出方法解决的百分比。
另一方面，UCB/LinUCB这个策略能否与kkkNN-LM相结合，用在API推荐中，具体来说把每个API都存储成一个向量，使用UCB来决定是否是依赖LM，还是使用kNN.

2.2 Learning Math Reasoning from Self-Sampled Correct and Partially-Correct Solutions

和自然语言一样，代码也存在着别名(Alias), 这意味着在生成某个问题的代码解决方案时，可能正确的不止一种。但是，目前在训练模型时，通常只会考虑一种正确解答，也就是<问题、代码>对。这篇工作期望在训练模型时，如果模型生成了正确但是与标准解答不一致的答案时，应当给予认可，而不是将其视为错误(损失)。

伪代码:
输入: 模型参数θ\thetaθ, 执行器E\EpsilonE, 一个训练样本(x,y∗,z∗)(x, y*, z*)(x,y∗,z∗), Buffer BBB for xxx
其中，xxx是自然语言描述， y∗y*y∗是正确的代码标签，z∗z*z∗是正确的输入输出对。BBB用来存储全部正确的代码。

if B is empty:
    B = B + {y*}
all_samples = sampleSolution(x, theta, B)  # 从模型中采样代码
for sample in all_samples:
    # execute sample with E 
    if iscorrect(sample) and sample is not duplicate in B:  # 保证B中不会出现重复的代码
        B = B + {sample}

# 更新模型参数theta， 更新的方式使用多个正确标签下的损失计算

上述伪代码中采样到完全正确的代码其实是比较困难的。因此，本文也利用了部分正确的代码。部分正确的含义是比较生成的代码和正确代码之间的变量快照。如果在前ttt行的代码快照与正确代码一致，则说明前ttt行代码的语义是一致的。因此可以将前ttt行代码与ttt行之后的正确代码进行拼接，从而生成正确的代码。

思考： 部分正确的代码似乎也是有用的，这篇论文是通过variable tracing的方式来衡量代码是否部分正确。这是否可以帮助A*算法来识别到离正确代码的距离呢？微软正好也做了预测Tracing的模型，是否可以一起使用，来不需要execute就可以避免一些完全不正确的代码出现。
同时，作者在汇报中也给出了两个chatgpt生成失败的例子，即让chatgpt生成不正确的代码。但是发现，chatgpt生成的代码依然是正确的，只是描述代码时说明是错误的。能否通过control 或者描述和代码一致性方法来避免出现这种情况呢？

3.1 CodeAttack: Code-Based Adversarial Attacks for Pre-trained Programming Language Models

这篇文章主要针对代码模型进行攻击，尽可能保持代码的语义一致性，同时让生成的性能下降。

  • 性能下降的约束:

Q(F(X))−Q(F(Xadv))≤δQ(F(X)) - Q(F(X_{adv})) \leq \delta Q(F(X))−Q(F(Xadv​))≤δ

F(X)F(X)F(X)表示任务输出，对于代码摘要是自然语言，对于代码翻译是新语言的代码. Q(⋅)Q(\cdot)Q(⋅)表示评估指标，比如对于摘要可以是BLUE.

  • 代码语义一致性约束:

∣∣Xadv−X∣∣<θ and Sim(Xadv,X)>ϵ||X_{adv} - X|| < \theta \text{ and } Sim(X_{adv}, X) > \epsilon ∣∣Xadv​−X∣∣<θ and Sim(Xadv​,X)>ϵ

生成攻击样本的步骤:

  1. 找到重要的token. 对于每一个token进行Mask, 对比模型输出的Logit outputs. 将逻辑输出与原始的逻辑输出进行对比，选择前kkk个差异最大的Mask词。这些词就当作重要token.
  2. 使用CodeBert(或者任意的Masked Langauge Model)来对重要Token生成替换。生成的Mask预测需要满足两个限制。
    • Mask预测的词和初始词最多只有一个子token不一致
    • Mask预测的词和初始词的Type一致。比如Mask的是Identifier, 生成的词也必须是Identifier类型
  3. 评估替换的代码是否满足性能下降约束 和 代码语义一致 约束，如果满足，返回替换的代码。
